## Codebase Patterns
(Reusable learnings discovered across wave-2 iterations. Keep this section short
and general — story-specific notes live in the iteration entries below. Patterns
inherited from wave 1 are NOT duplicated here; read the archive for those.)

- Terminal-game detection (US-049): a game is "terminal" iff `Game.status == 'COMPLETED'`. The runner sets this atomically with the `GameTerminated` event row inside `GamePersistence._persist_terminated_event`. Do NOT add new `event_type='GameTerminated'` queries — filter on `Game.status` instead. The legacy event-scan pattern is gone from `padrino.api` and `padrino.gauntlets` and should not be reintroduced.
- `Game.terminal_result` is a JSON column shaped `{winner, reason, day_terminated}`. The separate `Game.terminal_reason` column was dropped — reason lives inside the JSON. Read it via `game.terminal_result["winner"]`.
- Game-status string values are UPPERCASE to match the existing Gauntlet convention: `CREATED` → `RUNNING` → `COMPLETED`. Story copy showed lowercase as an illustrative example; the codebase uses uppercase.
- Atomic terminal-write seam: `_persist_terminated_event(persistence, stored, state, ranked, day_terminated)` writes the event row, `Game.status='COMPLETED'`, `Game.terminal_result`, and (when applicable) every rating update inside ONE `session.begin()`. Tests assert rollback by monkeypatching `update_ratings_for_game` to raise — the game row and event row must NOT be persisted.
- Seat-backfill seam: `_persist_roles_assigned(persistence, stored)` writes the `RolesAssigned` event row and every `game_seats` row in ONE txn. Requires `GamePersistence.agent_builds` to cover EVERY seat in the payload; partial mappings silently skip seat insertion (the event row still goes through) so callers driving the runner without a per-seat build mapping don't crash. Demo gauntlets and full integration tests always supply the complete map.
- The runner forbidden-imports firewall test (`test_game_runner_does_not_import_forbidden_modules` in `tests/runner/test_game_runner.py`) blocks `random`, `secrets`, `datetime`, `time` imports from `src/padrino/runner/game_runner.py`. If a story needs wall-clock at the runner level, factor the helper into a sibling module (e.g. `padrino.runner.persistence` or similar) and import the helper — don't import the stdlib module from `game_runner.py`.
- Secret resolution (US-050): `padrino.llm.secrets.resolve_secret(ref)` is the single seam that turns `ModelProvider.auth_secret_ref` strings into credentials. Supported schemes are `env:VAR_NAME` and `file:/abs/path` ONLY (no `vault:`, no relative paths, no `~` expansion). Raise `SecretResolutionError` on every failure mode — never return `None` or `""`. The adapter resolves ONCE at `__init__` and stashes the value on `self._auth_secret`; never re-resolve per request. The admin POST /model-providers route also resolves at creation so a bad ref 422s immediately instead of 401-ing at game time.
- The `padrino.llm.litellm_adapter` AST firewall test (`test_litellm_adapter_does_not_import_db_or_random`) checks `mod.startswith(forbidden)` for forbidden module names including `"secrets"`. The stdlib module is `secrets`; our project module is `padrino.llm.secrets`. Because the prefix check is anchored at the start of the dotted name, `padrino.llm.secrets` does NOT match the `secrets` prefix — safe to import. Future stdlib `secrets` (token generation, etc.) imports from the adapter would still be caught.
- Recorded-cassette tests (US-051): use `vcrpy` (committed dev dep, ignored by mypy via the `vcr.*` override). The `live_llm` marker is registered in `pyproject.toml` and default-skipped by the `pytest_collection_modifyitems` hook in `tests/conftest.py`. Opt-in is either `--live-llm` (CLI flag) or `-m live_llm` (marker selector). Cassette helper: `vcr.VCR(record_mode="once" if PADRINO_RECORD_LLM=="1" else "none", match_on=("method","scheme","host","path"), filter_headers=[...], before_record_request=_scrub_request, before_record_response=_scrub_response)`. Loose `match_on` lets cassettes ignore differing request bodies across model parameters. The audit test `test_cassettes_have_no_secret_shaped_substrings` scans every committed yaml for `sk-...`, `sk-ant-...`, and `Bearer <token>` patterns and is unmarked so it runs on every default invocation.
- Probing litellm's transport: vcrpy 8.x intercepts the async httpx transport via `vcr.stubs.httpcore_stubs` (the older `httpx_stubs` module was renamed). Litellm dispatches to provider-specific URLs which can be observed by monkeypatching `httpx.AsyncClient.send`. Confirmed URLs as of the wave-2 commit: openai→`api.openai.com/v1/chat/completions`, anthropic→`api.anthropic.com/v1/messages`, cerebras→`api.cerebras.ai/v1/chat/completions`, deepinfra→`api.deepinfra.com/v1/openai/chat/completions`, ollama (with `ollama/<model>` prefix, NOT `ollama_chat/`)→`localhost:11434/api/generate`. The ollama provider's `/api/generate` returns `{"response": "<text>", ...}` (not the `/api/chat` shape with `message.content`).
- Canonical per-role prompts (US-052): `padrino.llm.prompts.load_canonical(ruleset_id, role_family)` returns a frozen `PromptTemplate` with `system_prompt` text loaded from bundled markdown under `src/padrino/llm/prompts/<ruleset_id>/<role_family>.md`. `iter_canonical_prompts()` yields all four; `canonical_prompts_by_role()` projects them into a `Mapping[Role, str]` (already family-resolved) that `LiteLlmAdapter.system_prompts_by_role` consumes — the adapter does NOT import a ruleset module. Build-pinned non-canonical prompts route through the legacy single `system_prompt` arg.
- Subpackage-vs-import namespace collision: `padrino.llm.prompts` is a package containing the `mini7_v1` ruleset directory; doing `from padrino.core.rulesets import mini7_v1` inside `prompts/__init__.py` makes the LOCAL `mini7_v1` attribute get clobbered the first time `importlib.resources.files("padrino.llm.prompts.mini7_v1")` imports the subpackage (Python rebinds the parent module's attribute). Workaround applied: alias the ruleset import (`as _mini7_ruleset`). If a future story adds another `prompts/<ruleset_id>/` directory, use the same alias pattern.
- `role_family_breakdown` shape (`Mapping[role_family_value, Mapping[str, float]]`) committed in `padrino.leaderboards.service._per_ab_role_family_breakdown`: keys are the `RoleFamily` enum values (DECEPTIVE / INVESTIGATIVE / PROTECTIVE / VANILLA_TOWN); each bucket carries `{games, wins, draws, losses, win_rate}` as floats (win_rate is `wins/games`, 0.0 when games == 0). Seats with a `role` string not parseable as `Role(...)` are skipped silently — defensive against historical rows the leaderboard route should never crash on.
- Hatch wheel packaging for non-Python data: `[tool.hatch.build.targets.wheel].artifacts = ["src/padrino/llm/prompts/**/*.md"]` is the right knob to force-include `.md` files alongside Python sources. `importlib.resources.files("padrino.llm.prompts.<ruleset>").joinpath("<file>.md").read_text()` works in src-mode immediately because the directory is on sys.path; the `artifacts` config matters when building the wheel for distribution.
- Retry / backoff seam (US-053): `padrino.llm.retry.with_retry(call, policy, *, sleeper, rng)` is the one place LLM provider calls go through exponential backoff. Both time and randomness are injected so adapter tests pin them; `sleeper` defaults to `asyncio.sleep` on the adapter and `rng` is a per-call `SeededRng(f'{phase}:{player_id}:{model}')` so replays reproduce the schedule. Exhaustion raises `RetryExhausted` carrying the last error; the adapter's `_call_model` catches it and returns `AdapterResult(status='provider_error', failure=LlmCallFailed(...))`. The adapter's `complete()` promotes the final status to `'exhausted'` (new `AdapterStatus` literal) when either the no-fallback path or the fallback path saw exhaustion. Tick.py inspects `result.status == 'exhausted'` and emits ONE `ActionTimedOut` event with `payload.reason='llm_exhausted'` plus `error_kind / error_message / attempts` (matches the new `llm_calls.error_kind / error_message` columns from migration 0006). Existing `'both_failed'` is preserved for the two-non-retryable case so legacy tests didn't have to flip.
- litellm exception classes must be imported from `litellm.exceptions` (NOT `import litellm; litellm.RateLimitError`) under mypy strict. The `[[tool.mypy.overrides]] module = ["litellm.*"]` block sets `ignore_missing_imports = true` but does NOT suppress `attr-defined` errors on the top-level `litellm` module's lazy exception re-exports. Use `from litellm.exceptions import RateLimitError, APIConnectionError, InternalServerError, BadRequestError, AuthenticationError` instead.
- `EventLog.events` returns a tuple of `StoredEvent` Pydantic models, not dicts. Access body fields via `event.body["event_type"]` (and `.body["payload"]`), not `event.get("event_type")`. The `body` is a dict, but the wrapper is frozen.
- Async scheduler seam (US-054): `padrino.runner.scheduler.run_scheduler(session_factory, *, concurrency, stop_event, adapter_factory, game_executor, semaphore, options, clock, sleeper)` is the single drain loop for `Gauntlet.status='PENDING'` rows. Status transitions: PENDING → RUNNING (claim + heartbeat) → COMPLETED (set `completed_at`, clear `heartbeat_at`). Crash recovery runs once at startup; stale-RUNNING means `heartbeat_at < now - heartbeat_interval_s * stale_factor` OR `heartbeat_at IS NULL`. Tests inject `game_executor` so they don't need to run full mini7 games. The `padrino.runner.game_runner` purity-firewall test is scoped only to `game_runner.py`; sibling impure modules in `padrino.runner/` are free to import `asyncio`, `datetime`, `time`.
- Cross-dialect datetime comparisons: SQLite drops tzinfo on read, Postgres preserves it. Use `padrino.db.repositories.gauntlets._aware(dt)` (or copy the pattern: `dt if dt.tzinfo is not None else dt.replace(tzinfo=UTC)`) before any comparison between a DB-read datetime and `datetime.now(UTC)`. The pattern will matter once US-057 lands a full Postgres test suite.
- Gauntlet status convention is uppercase `PENDING` / `RUNNING` / `COMPLETED`. `gauntlets_repo.create()` defaults to `'PENDING'`. The legacy `'QUEUED'` literal in the v1 API route was an outlier and is gone as of US-054. Any new states (e.g. `'FAILED'`) should follow the same uppercase convention.
- API pagination seam (US-055): `padrino.api.pagination.CursorPage[T]` is the standard envelope for every list endpoint — `{items, next_cursor, total_estimate}`. Cursors are opaque base64-encoded JSON: `encode_cursor(created_at, id)` for keyset pagination over time-ordered DB lists; `encode_index_cursor(n)` for in-memory ordering (the leaderboard). `paginate_keyset(session, stmt, *, created_at_col, id_col, limit, cursor)` is the canonical helper — it applies `(created_at, id) > (cursor_dt, cursor_id)` WHERE, ORDER BY both columns, and `LIMIT n+1` to detect a next page. Limit bounds are `MIN_LIMIT=1, DEFAULT_LIMIT=50, MAX_LIMIT=200`. Tampered cursors raise `InvalidCursorError`; routes call `invalid_cursor_error()` to surface a 400 `{detail: 'invalid_cursor'}`.
- Strict query params (US-055): unknown query parameters return 422 via the `Annotated[QueryModel, Query()]` FastAPI 0.115+ pattern where `QueryModel` is a `BaseModel` with `model_config = ConfigDict(extra='forbid')`. Per-route limit field uses `Field(default=DEFAULT_LIMIT, ge=MIN_LIMIT, le=MAX_LIMIT)` so out-of-range values 422 at validation, before the handler runs. New list endpoints follow this pattern uniformly.
- `games.created_at` was added in migration 0008 (US-055) and is the canonical ordering column for `GET /games`. The pre-existing `started_at` is "when the runner began ticking", which is `NULL` for unstarted games and therefore unsuitable as a sort key. Existing rows backfill `created_at` from `COALESCE(started_at, completed_at, CURRENT_TIMESTAMP)` before the NOT NULL constraint is added; rely on `Game.created_at` whenever you need an insertion-order key on games.
- Leaderboard route shape (US-055): `GET /leagues/{id}/leaderboard` keeps its existing `entries` field for back-compat and just adds a top-level `next_cursor` — it does NOT return a `CursorPage[T]` envelope, because the payload also carries `leaderboard_id / ruleset_id / prompt_version / rating_model` that don't fit `CursorPage`'s shape. Pagination is index-based over the already-sorted in-memory entries; cursor uses `encode_index_cursor`. `compute_leaderboard(...)` now accepts an optional `gauntlet_id` filter that narrows `_terminal_games_in_league` to one bracket.
- API-key auth seam (US-056): authentication is **opt-in** via `create_app(auth_required=True, admin_token=..., rate_limiter=...)`. With `auth_required=False` (the default) requests with no credentials get a *synthetic admin* `ApiKeyContext(id=None, scopes={admin}, via_admin_token=False)` — this keeps every existing route + test green without modification. `padrino.api.auth.get_auth_context(request, session)` is the single resolution point; per-route `Depends(require_admin)` / `Depends(require_read)` enforce scopes on top of it. Scope policy: `admin` scope satisfies any required set, otherwise an intersection is required. New write routes must add `dependencies=[Depends(require_admin)]`; new read routes use `Depends(require_read)`.
- Rate-limit seam (US-056): `padrino.api.auth.RateLimiter` is an in-process sliding-window counter (per-key deque of monotonic timestamps, 60-second window). `clock` is injectable for tests — substitute a `_FakeClock` callable returning a mutable float. The settings ceilings are split across three fields (`padrino_rate_limit_admin_per_minute`, `padrino_rate_limit_submitter_per_minute`, `padrino_rate_limit_spectator_per_minute`); the story's single-field shape was infeasible because the three defaults differ. Synthetic admin contexts and X-Padrino-Admin-Token shim contexts skip rate-limiting entirely — only api-key Bearer requests count against the bucket. Tests that need to exercise the limiter monkeypatch `padrino.api.auth._limit_for_scopes` to a low ceiling (e.g. 2).
- Raw-key handling convention (US-056): raw keys live ONLY in the POST /admin/keys response body (and once-only in the `--with-admin-key` bootstrap CLI later in US-058). Persistence is `sha256(raw_key)` in `api_keys.key_hash` UNIQUE + a 6-char `key_prefix` for display. The raw key format is `pk_<urlsafe-base64-32>` (≈ 46 chars); `api_keys_repo.hash_api_key(raw)` is the single place where the digest is computed. `hmac.compare_digest` is used only for the back-compat X-Padrino-Admin-Token check; sha256 lookups go through the UNIQUE index in O(log n) — the constant-time property comes from the DB layer, not Python. Any future endpoint that returns key metadata must NEVER expose `key_hash` or any `raw_key`-like value.
- Dialect portability (US-057): the `padrino.db.base.create_engine(url, *, echo, pool_size, max_overflow)` factory is the only place that branches on URL scheme. `postgresql+asyncpg://` accepts `pool_size` / `max_overflow` (forwarded to `create_async_engine`); `sqlite+aiosqlite://` ignores them (aiosqlite + StaticPool don't honor a server pool) AND installs the `PRAGMA foreign_keys=ON` connect listener. Postgres has FK enforcement on by default — no driver-specific path needed beyond engine construction. Use only the driver-agnostic `sa.JSON` / `sa.Uuid` / `sa.DateTime(timezone=True)` types in models and migrations; the static `tests/db/test_dialect_portability.py::test_no_dialect_specific_json_in_models` test will fail if a new model leaks `dialects.postgresql.JSONB` or `dialects.sqlite.JSON`.
- Postgres test infra (US-057): `testcontainers>=4.7` + `psycopg[binary]>=3.2` are dev deps. The `[postgres]` / `[postgresql]` extras do NOT exist on testcontainers 4.x — `from testcontainers.postgres import PostgresContainer` imports unconditionally with just the bare `testcontainers` package. Use `PostgresContainer("postgres:17-alpine", driver="asyncpg")` to get an `asyncpg`-flavored connection URL straight from `pg.get_connection_url()`. `_sync_url(url)` (in `tests/db/test_migrations_all_dialects.py`) translates `postgresql+asyncpg://` → `postgresql+psycopg://` for the alembic env / sync inspector path. Sessions are session-scoped so the container starts once per `pytest` invocation; per-test cleanup drops every Padrino table + `alembic_version` so each migration test starts truly empty.
- Postgres-marker auto-skip (US-057): `tests/conftest.py::pytest_collection_modifyitems` now also skips items tagged `postgres` when `docker info` doesn't return within 5 s. The check uses `shutil.which("docker")` + a `subprocess.run` with `timeout=5, check=False` so collection stays fast on dev machines without Docker. Tests written for the marker should assume the container is running; never embed availability checks inside individual test bodies.
- Driving alembic from inside an `asyncio.run`-wrapped CLI (US-058): `alembic.command.upgrade(cfg, 'head')` is a sync API but Padrino's `migrations/env.py` calls `asyncio.run(_run_async_migrations())` internally. If the caller is already inside an event loop (e.g. a Typer command that wraps its work in `asyncio.run(...)`), driving the upgrade synchronously raises `RuntimeError: asyncio.run() cannot be called from a running event loop`. The fix is to dispatch the sync alembic call via `await asyncio.to_thread(_run_migrations, db_url)` so the nested `asyncio.run` lives on a worker thread with no enclosing loop. The same pattern applies to any future CLI / scheduler-driven migration touchpoint.
- Building an `alembic.Config` programmatically (US-058): `AlembicConfig()` with no path is the right shape when the migration directory lives inside the installed package. Set `script_location` to the absolute path of the package's migrations dir (`Path(padrino.__file__).parent / 'db' / 'migrations'`) and `sqlalchemy.url` to the target URL — Padrino's `env.py` reads `PADRINO_DB_URL` from env, so set that too for the duration of the upgrade. Don't rely on a co-located `alembic.ini` for installed deployments; the repo's `alembic.ini` is dev-only.
- Pydantic strict YAML schemas (US-058): `ConfigDict(extra='forbid', frozen=True)` on `ProviderSpec` plus `extra='forbid'` on the wrapping `ProvidersFile` is the right shape for a YAML config file — unknown keys raise `ValidationError` at `model_validate(...)` time. Wrap the call site in a `BootstrapError(step, str(exc))` to surface the failing field through the CLI's exit-code path. `yaml.safe_load` returns `Any`; cast/validate explicitly before consuming.
- Bootstrap idempotency convention (US-058): every seeding step in `padrino.bootstrap` reports one of three statuses — `ok` (action taken), `skipped` (rows already existed), or `failed` (`BootstrapError`). Migration step always reports `ok` (alembic handles idempotency internally — re-running upgrade-head is a no-op but the step still runs). Admin-key minting is NOT idempotent: each `--with-admin-key` invocation creates a fresh key. This is intentional — operators may want to issue multiple admin keys; the previous keys remain valid until explicitly disabled.
- Prometheus metrics seam (US-059): the canonical surface is `padrino.observability.metrics` (the DB-backed `padrino metrics` CLI helpers moved to `padrino.observability.summary`). Instruments live on a dedicated `REGISTRY` (a fresh `CollectorRegistry`, NOT the prometheus_client process-global) so tests can `reset_metrics()` without colliding with third-party libs. `record_llm_call(model_id, status, latency_ms)`, `record_phase_duration(ruleset, phase_kind, duration_s)`, `record_game_completed(outcome, ruleset)`, `record_invalid_action(reason)` are the canonical helpers — call sites colocate the helper with the existing US-048 `_logger.info(EVENT_*)` calls. `split_litellm_model_id("cerebras/zai-glm-4.7")` → `("cerebras", "zai-glm-4.7")`; the DeepInfra triple-slash form (`deepinfra/deepseek-ai/DeepSeek-V4-Flash`) keeps everything after the first slash as the model name (matches the litellm provider/model convention).
- Wall-clock seam for the runner (US-059): `padrino.observability.timing.time_phase(ruleset, phase_kind)` is a `contextlib.contextmanager` over `time.monotonic()`. `game_runner.py` must import this helper rather than `time` directly because the purity firewall (`test_game_runner_does_not_import_forbidden_modules`) rejects `random/secrets/datetime/time` AST imports — a re-export through `padrino.observability.timing` is safe because the firewall checks the FIRST dotted-name segment (`padrino.*` passes). Use this helper whenever a future story needs to time a block inside `game_runner.py`.
- `/metrics` endpoint shape (US-059): `padrino.api.app.create_app(metrics_require_auth=...)` registers `GET /metrics` returning `text/plain; version=0.0.4; charset=utf-8` (Prometheus exposition spec via `prometheus_client.CONTENT_TYPE_LATEST`). The middleware `metrics_middleware` increments `padrino_api_requests_total{route,method,status}` using the FastAPI route template — `request.scope['route'].path` — so dynamic ids stay out of the cardinality. Fallback to `request.url.path` for routes that haven't matched (404s, mounted ASGI apps). `Settings.padrino_metrics_require_auth: bool = False` defaults to open scraping; flipping it routes through `Depends(require_read)` so the spectator scope (or admin) is required.
- `prometheus_client.Counter._metrics.clear()` is the documented way to reset per-label-set children for a test (`prometheus_client` doesn't ship a public `.reset()`); a private API but the upstream test suite uses it the same way. `Gauge.set(0)` works as expected. Run `reset_metrics()` from an autouse fixture in any test module that calls a `record_*` helper — collectors are module-level and otherwise persist across tests.
- Scheduler readiness seam (US-060): two distinct heartbeat surfaces — `gauntlets.heartbeat_at` (US-054) is per-gauntlet liveness used by crash recovery (stale RUNNING row → reset to PENDING); `scheduler_heartbeats(worker_id PK, beat_at)` (US-060, migration 0010) is per-worker liveness used by the `/healthz/scheduler` readiness probe. Worker writes its row every tick via `padrino.db.repositories.scheduler_heartbeats.upsert(session, worker_id, beat_at)`. `default_worker_id()` = `f'{socket.gethostname()}:{os.getpid()}'`; injectable via `run_scheduler(..., worker_id=...)` for tests. The `/healthz/scheduler` endpoint reads `MAX(beat_at)` so multiple workers in the table all contribute — one stale worker doesn't degrade status as long as another is fresh. Status thresholds: down if `last_heartbeat_at IS NULL` OR age > 30 s; degraded if `oldest_pending_age_s > 60`; otherwise ok. Endpoint is unauthenticated (matches `/healthz`'s scrape-pattern policy).
- Gauntlet queue-stats helpers (US-060): `gauntlets_repo.count_by_status(session, status)` returns the row count for one status; `gauntlets_repo.oldest_pending_created_at(session)` returns `MIN(created_at) WHERE status='PENDING'` (timezone-coerced via `_aware()` for the SQLite naive-datetime drop). These are the canonical aggregates for health/queue endpoints — prefer them over hand-rolled `select(func.count())` at call sites so the timezone handling stays in one place.
- `EXPECTED_TABLES` in `tests/db/test_migrations_all_dialects.py` is BOTH an assertion target AND the source of the postgres downgrade-cleanup loop (`for table in (*EXPECTED_TABLES, "alembic_version"): DROP TABLE IF EXISTS ...`). Any new migration that creates a table MUST add the name to this set or the next postgres test will fail with `DuplicateTableError` because the cleanup loop won't drop it. `tests/db/test_migrations.py`'s SQLite-only `EXPECTED_TABLES` is `issubset`-only and doesn't need updating, but the all-dialects file is exclusive.
- Game-export bundle seam (US-061): `padrino.export.bundle.export_game(session, game_id, *, signer=None)` is the single point for emitting a federated-leaderboard artifact. The bundle is a Pydantic `GameBundle` (schema_version='padrino.export.v1') containing league/gauntlet/game ids, ruleset_id, seed, terminal_result, tip_hash, per-seat `AgentBuildInfo` (display_name + prompt_version + provider name + model_name + model_version — NEVER `auth_secret_ref`), per-seat `GameSeatInfo`, the full event log as `EventEnvelope` rows, plus optional `signer_fingerprint`/`sig`. `canonical_bundle_bytes(bundle)` is the canonical-JSON byte representation used for signing — `bundle.model_dump(mode='json', exclude={'sig'})` → `canonical_dumps(...)`. Two exports of the same game are byte-identical because `canonical_dumps` sorts keys and rejects floats/datetimes; pydantic's `mode='json'` turns UUIDs into strings before they hit the encoder. Hash-chain reconstruction (`_event_body`): the body that the engine originally hashed is `{event_type, sequence, phase, visibility, actor_player_id, payload}` — `event_hash`, `prev_event_hash`, and `created_at` are excluded by `padrino.core.engine.hashing._EXCLUDED_KEYS`. `_verify_chain(events)` re-folds the bodies through a fresh `EventLog` via `padrino.core.engine.replay.replay_event_log` and returns the head_hash; tampering raises `ReplayHashMismatchError`. Defense-in-depth `assert_bundle_payload_safe(events)` scans payloads for `EXPORT_FORBIDDEN_PAYLOAD_KEYS` (model_id / model_name / provider / rating / openskill_* / gauntlet_clone_index / auth_secret_ref / api_key / authorization — NOT `role`/`faction`, which legitimately appear in RolesAssigned/PlayerEliminated payloads); these keys should never appear in engine-emitted payloads, so a hit means a bug or tampered DB.
- Ed25519 signing convention (US-061): `cryptography>=42.0` is a runtime dep. `padrino.export.bundle.Ed25519Signer.from_env('PADRINO_EXPORT_PRIVATE_KEY')` decodes a base64 (urlsafe) 32-byte SEED (not the SPKI public-key format) via `Ed25519PrivateKey.from_private_bytes(seed)`. `signer.fingerprint` is `sha256(public_key_bytes)[:32]` (first 32 hex chars). Signatures are over `canonical_bundle_bytes` AFTER `signer_fingerprint` is set — this binds the signed bytes to a single identity, so `verify_bundle_signature(bundle, pub_b64)` also checks that the supplied public key's fingerprint matches `bundle.signer_fingerprint`. `sig` is base64 (urlsafe, padded) of the 64-byte signature.
- Ingestion seam (US-062): `POST /ingest/game` is the single federated-submission entrypoint. Submitter scope (or admin) is required via `require_scopes(SCOPE_ADMIN, SCOPE_SUBMITTER)`. Pipeline: 10 MB body cap (413) → `GameBundle.model_validate_json` (422 `invalid_bundle`) → duplicate-`game_id` short-circuit returning 200 `{already_ingested: true}` BEFORE replay → `assert_bundle_payload_safe` (422 `unsafe_payload`) → `padrino.export.bundle.verify_chain` + tip equality (422 `hash_chain_mismatch`) → signature verification only when `bundle.sig` is present; a signed bundle without a registered `api_keys.submission_public_key` is 422 `signature_unverifiable`, a mismatched signature is 422 `signature_mismatch`. Persistence row in `ingested_games` carries `verification_status='verified'` (signed + verified) or `'unverified'` (no sig present). `submitter_key_id` is the api_keys FK; `league_id` / `gauntlet_id` are opaque strings, NOT FKs, because the originating identifiers belong to a different deployment.
- Public read-API seam (US-063): `/public/*` routes live in `padrino.api.routes.public` and share one `require_public_read` dep that flips between "anonymous" and "spectator/admin required" based on `Settings.padrino_public_leaderboard_anonymous`. When the flag is True the dep short-circuits with a synthetic spectator context BEFORE touching `get_auth_context`, so anonymous requests don't hit the api_keys lookup at all. When the flag is False the dep delegates to `get_auth_context` and then re-checks scopes — the inner dep already 401s on missing creds (when `auth_required=True`) and the outer dep adds the 403 path for valid keys with the wrong scope.
- Public-leaderboard aggregation (US-063): `padrino.ratings.public_leaderboard.compute_public_leaderboard(session, *, ruleset_id, gauntlet_id=None)` rolls openskill PlackettLuce across every `ingested_games` row matching the filter, keyed by an `entity_id = sha256(display_name|model_provider|model_name|model_version|prompt_version)[:32]` tuple lifted directly off `bundle.agent_builds[*]`. Cache is a process-local dict keyed by `(ruleset_id, gauntlet_id, max(ingested_games.created_at).isoformat())` — a fresh submission bumps the max-created_at so the next call misses the cache and recomputes; older callers holding the prior tag are simply ignored. `reset_cache()` is exported for tests.

The adapter's two new init kwargs are `retry_policy: RetryPolicy | None = None` and `sleeper: Callable | None = None`. - **Learnings for future iterations:**
  - Acceptance-criterion interpretation: "runs a demo gauntlet via the API" → tested as `docker compose exec api padrino demo-gauntlet` (engine smoke through the image) rather than a true HTTP `POST /gauntlets`. A pure-HTTP gauntlet POST needs an `AgentBuild` row + `ModelConfig` + `Prompt` already seeded, which bootstrap doesn't create. US-068's localhost smoke harness is expected to layer the full HTTP path on top of US-064; if a future story needs the HTTP variant earlier, extend bootstrap (or add a `padrino seed-demo-build` subcommand) so the smoke can POST against a ready-to-run gauntlet without operator setup.
  - `padrino serve` needs an explicit session_factory: `create_app()` defaults `app.state.session_factory=None` and `get_session_factory` 503s. Without wiring an engine from `Settings.padrino_db_url`, the served app's `/healthz/scheduler`, `/leagues`, `/games`, etc. all return 503 in a real deployment — this only didn't show up before because tests always pass their own factory and no production-grade serve invocation was ever exercised. Always pair a CLI `serve` with explicit `session_factory=` construction.
  - typer `envvar=` parameter: every `--db-url` style option should set `envvar='PADRINO_DB_URL'` so containers can drop the explicit flag. Typer reads the env var BEFORE applying the default literal, so the precedence is `--db-url > $PADRINO_DB_URL > literal default`. Adding `envvar=` is backwards-compatible (no behavior change when the env var is unset).
  - compose `service_completed_successfully` is the right gate for one-shot init containers — pair with `restart: 'no'` on the init service so it doesn't loop. Requires compose v2.20+; document the minimum in `docker-compose.yml`'s header when a future story tightens the requirement.
  - Docker-marker policy split: `live_llm` and `docker` are STRICTLY opt-in (require `-m <marker>`); `postgres` auto-runs whenever docker is reachable. The split comes from per-test cost: postgres uses one shared container so the marginal cost per test is tiny, but `docker` builds a fresh image per fixture so a single test can take minutes. When adding a new marker, decide cost upfront — copy the two-tier pattern for expensive markers and the one-tier `not-docker → skip` pattern for cheap ones.
