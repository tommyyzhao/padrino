## Codebase Patterns
- Scheduler/async DB tests on the in-memory SQLite engine (`sqlite+aiosqlite:///:memory:`, StaticPool = ONE shared connection) are sensitive to poll ORDER. A status-watcher coroutine that polls `gauntlets_repo.get` in a tight loop must `await asyncio.sleep(interval)` FIRST and read AFTER — reading first then sleeping starves the scheduler's drive/write coroutines of the single shared connection and the gauntlet never finalizes (the run hangs). Keep the cooperative `for _ in range(N): await sleep; read; if done: break` shape; raise N for a generous deterministic budget instead of converting to an unbounded `while True` (unbounded + read-first = deadlock under the StaticPool). Never `stop.set()` unconditionally before the target status is observed — that cuts a slow-but-correct run short and makes the final status assertion flaky. See `_drive_scheduler_until` in `tests/runner/test_scheduler.py`.
- Hermetic Playwright e2e for a public-surface page: intercept every `/public/*` call the page makes with `page.route('**/public/<path>*', ...)`, gating on `route.request().resourceType()` in `('fetch','xhr')` and `route.continue()` otherwise (so document/asset loads pass through). This is the lobby/recap/watch pattern; `games.spec.ts` and `leaderboard.spec.ts` now follow it so they no longer depend on whatever the smoke harness happens to seed into the broadcast/ingest indexes (which post-US-112 excludes unverified ingests, making row presence state-dependent). The replay scrub path needs `/public/games/{id}/events*` mocked with >=2 distinct `phase` values and a terminal `GameTerminated` frame so the phase pills + next/prev disabled states are deterministic.
- Per-test Postgres isolation for migration cycle tests: a session-scoped container shared across tests must NOT share one mutable database (leftover objects break the next `upgrade head`). Create a UNIQUE throwaway database per test (`CREATE DATABASE padrino_mig_<uuid>` on an AUTOCOMMIT connection to the maintenance DB — CREATE/DROP DATABASE cannot run in a transaction), point `PADRINO_DB_URL` at it, and `DROP DATABASE` (after `pg_terminate_backend` of lingering sessions) in teardown. Swap the URL path with `urlsplit`/`urlunsplit`. See `_with_database` + the `postgres_db_url` fixture in `tests/db/test_migrations_all_dialects.py`.
- CLI logging is configured ONCE in the Typer `@app.callback()` (`_configure_logging_callback` in `cli.py`), which runs for every subcommand. To honor an env-driven setting across serve/scheduler/bootstrap, read it there from `get_settings().padrino_log_level` rather than per-command. `configure_logging` is idempotent (`_configured` flag), so the callback firing on every invocation is safe. - Pure-planner + impure-executor split: a pure function under `core/` or `db/retention.py` returns a plan (list of ids); the impure executor (`db/retention_executor.py`) loads the projection, calls the planner with an injected `now`, and applies the plan inside `async with session_factory() as session, session.begin()`. ## Codebase Patterns

- `create_app(public_surface_only=True)` (or `PADRINO_PUBLIC_SURFACE_ONLY=true`) builds the internet-facing process: only the public router + health router are mounted; private routers and `/metrics` are NOT registered, so private prefixes 404 (route absent), never 401/403. The flag resolves like the other `create_app` knobs: explicit kwarg wins, else `settings.padrino_public_surface_only`. Note `scheduled_gauntlets_router` carries BOTH private CRUD and one public route (`/public/scheduled-gauntlets`); gating the whole router (per AC) drops that public route too — the must-work public surface is the enumerated set (`/public/live`, `/public/recent`, `/public/ladder`, `/public/games/{id}/live`).
- `src/padrino/public/` is the new module for Wave 7 public surface. Start each new public module here.
- The public spectator website (`web/dashboard`) must call ONLY `/public/*` endpoints from any nav- or URL-reachable page (it ships against the public-surface-only API). To get a league_id without the private `/gauntlets`, chain public reads: `/public/recent` -> `GET /public/games/{id}.league_id`. The e2e smoke spawn sets `PADRINO_PUBLIC_LEADERBOARD_ANONYMOUS=true` (`tests/e2e/global-setup.ts`) so keyless `/public/*` reads work; the smoke harness does NOT seed broadcastable RECENT games, so `/public/recent` + `/public/live` are empty there (only `/public/ladder` has data from the gauntlet's ratings).
- `padrino.core.spectator_projection.project_event_for_spectator` is the identity-blind filter for PUBLIC events; always build on top of it rather than reimplementing visibility filtering.
- `padrino.core.observation_privacy.FORBIDDEN_PAYLOAD_KEYS` is the canonical deny list for identity/role/faction leaks — shared across spectator, ranked-observation, and public_event_v1.
- Golden-file tests live in `tests/public/fixtures/` as JSON. The test loads the file and diffs computed output against stored expected — any schema drift fails the test until the golden file is updated consciously.
- `gauntlets.claim_oldest_pending` is the single scheduler claim seam (called from `runner/scheduler.py`). It applies `with_for_update(skip_locked=True)` ONLY when `session.get_bind().dialect.name == "postgresql"`, making horizontal scaling safe on Postgres (two replicas never double-claim) while leaving SQLite single-writer. For any new dialect-conditional SQL in a repo, branch on `session.get_bind().dialect.name`. To test cross-replica DB concurrency, give each claimer its own `factory()` session + `commit()` and `asyncio.gather` them (a shared session serializes and proves nothing); reuse the testcontainers `postgres_url`/`postgres_engine` fixtures from `tests/db/test_dialect_portability.py` under `@pytest.mark.postgres`.
- `padrino.observability.alerts.AlertNotifier` is the canonical operational-alert seam: `await notifier.fire(key, **details)` is fire-once-per-transition (de-duped via a per-key active flag) and `notifier.resolve(key)` re-arms it; it always logs `alert.fired` and POSTs JSON to `padrino_alert_webhook_url` when set. Build it with `build_alert_notifier(settings)`. The matchmaking tick (`run_continuous_matchmaking_tick`, optional `notifier=` param, threaded from `scheduler/bootstrap.py` + `cli.py`) is where spend-cap / stale-heartbeat / guard-unavailable / admission-denial-streak conditions are wired.
