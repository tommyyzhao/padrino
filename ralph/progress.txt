## Codebase Patterns
(Reusable learnings discovered across iterations. Keep this section short and general — story-specific notes live in the iteration entries below.)

- Project layout: `src/` layout under `src/padrino/`. Tests live in `tests/`. Use `from padrino.X import Y`.
- Quality gates: `uv run ruff check . && uv run ruff format --check . && uv run mypy src tests && uv run pytest -m "not integration"`. CI runs the same.
- Pure core: nothing under `src/padrino/core/` may import from `padrino.db`, `padrino.llm`, `padrino.api`, `padrino.runner`, or call wall-clock / network / random. Inject impure dependencies via constructor.
- Hash chain: `event_hash = sha256(prev_event_hash + canonical_json(event_without_hash_and_timestamp))`. Timestamps and the hash field itself are excluded from the hash input.
- Mafia chat is never parsed for mechanics — `action` field only. Tests must enforce this.
- Ranked observations never include model identity, ratings, gauntlet clone index, or other games' transcripts.
- Use `from __future__ import annotations` at the top of every module. Pydantic v2 models with `model_config = ConfigDict(frozen=True)` for state/event types.

- Runner emit pattern: `_emit(body, state, event_log)` returns the next `GameState` after (1) auto-assigning `body["sequence"] = len(event_log.events)`, (2) `event_log.append(body)`, (3) `EventAdapter.validate_python(body)`, (4) `apply_event(state, event)`. Every orchestrator-emitted event must go through one helper like this so state and the hash chain stay in lockstep; never call `event_log.append` and `apply_event` separately.
- Submission-event emission dispatches on `current_phase.kind`, not `action.type`. The runner is the legal-actions firewall; resolvers downstream trust their inputs and do not re-enforce phase eligibility.
- pydantic-settings: use `Settings(_env_file=None)` in tests so monkeypatch controls env vars without the on-disk .env interfering.
- structlog configure_logging uses a module-level `_configured` flag for idempotency.
- Mypy strict rejects module-level `def f(x: T) -> U` as a satisfier of a Protocol method (positional-only/staticmethod mismatch). Expose ruleset behaviour as `Final[dict[K, V]]` attributes (e.g. `ROLE_FACTIONS`) and have engine code read the mapping directly. Optional `*_for(...)` wrappers around the dict are fine for ergonomics, but the Protocol contract should be over the dict.

- Hypothesis stateful machines that terminate must NOT use `@precondition(lambda self: not self._done)` on the only rule — Hypothesis errors out with "no progress can be made" once the precondition stays False. Instead drop the precondition and have the rule body early-return when terminated. Bind `MyMachine.TestCase.settings = hypothesis.settings(max_examples=..., stateful_step_count=..., deadline=None)` and re-export `MyMachine.TestCase` as `TestSomething` for pytest discovery. Per-step variable-count draws use `data=st.data()` in `@rule(...)` and `data.draw(strategy)` in the body.

- DB layer (`src/padrino/db/`) lives outside `core/` and is therefore allowed to use wall-clock (`datetime.now(UTC)`) for `created_at` defaults — the pure-core hard rule is scoped to `src/padrino/core/` only. SQLite engines MUST attach a `PRAGMA foreign_keys=ON` connect-event listener on `engine.sync_engine` to enforce FKs; route every async-engine construction through `padrino.db.base.create_engine` so this is automatic. SQLite drops `tzinfo` on re-read even when the column is `DateTime(timezone=True)` — assert on `Model.__table__.c[col].type.timezone is True` or on the in-session pre-commit value, never on a refetched row's tzinfo.

- Alembic migrations live at `src/padrino/db/migrations/` with `script_location` configured in `alembic.ini` at repo root. The migrations dir is mypy-excluded and ruff-allowed for `E402` / `F401`. `env.py` reads `PADRINO_DB_URL` directly from `os.environ` (falling back to `Settings(_env_file=None).padrino_db_url`) so callers can override per-invocation without rewriting `alembic.ini`. The async engine is constructed via `padrino.db.base.create_engine` so SQLite FK enforcement stays consistent with the runtime path. `render_as_batch=True` is enabled for SQLite to support future ALTER-style operations. `alembic.ini` should set `path_separator = os` to silence the 1.16+ deprecation warning about `prepend_sys_path` splitting.

## 2026-05-14T00:45 - US-004
- Implemented `src/padrino/core/engine/rng.py`: `SeededRng(seed: str | bytes)` with stateful counter-based expansion via `sha256(seed_bytes + counter.to_bytes(8, "big"))`. Methods: `next_bytes`, `randbelow` (mask-and-reject for unbiased uniform), `shuffle` (Fisher-Yates returning a new list), `choice`.
- `tests/core/test_rng.py`: 23 tests covering determinism across instances, range bounds, statefulness, deterministic shuffle/choice, immutable inputs, and an AST assertion that `rng.py` does not import `random` or `secrets`.
- Files changed: `src/padrino/core/engine/rng.py`, `tests/core/test_rng.py`
- **Learnings for future iterations:**
  - Ruff `RUF023` requires `__slots__` tuple entries to be sorted (natural order). Easy to miss when slots are listed in declaration order.
  - For unbiased `randbelow(n)`, use `bit_length = (n - 1).bit_length()`, draw `ceil(bit_length / 8)` bytes, mask high bits to `(1 << bit_length) - 1`, then reject values `>= n`. Rejection probability is bounded at 50% per draw.
  - `SeededRng` is stateful: each `next_bytes`/`randbelow`/`shuffle`/`choice` advances the internal counter. Downstream callers should construct a fresh `SeededRng` per logical purpose (e.g., `SeededRng(sha256(b"roles" + game_seed))` for role assignment) to avoid coupling unrelated draws.

- **Learnings for future iterations:**
  - SQLite does NOT enforce foreign keys by default — every async engine pointed at `sqlite+aiosqlite://...` needs a `PRAGMA foreign_keys=ON` connect-event listener wired to `engine.sync_engine`. Tests that assert `IntegrityError` on FK violations will silently pass-without-the-error if this is missing, which is a foot-gun. `padrino.db.base.create_engine` is the single seam that should attach this; future stories (alembic env, integration tests, app factory) should always construct engines via this helper rather than calling `create_async_engine` directly.
  - SQLite drops `tzinfo` on read — even when the column is declared `DateTime(timezone=True)`, the value round-tripped through aiosqlite comes back as a naive datetime. Don't write tests that assert `row.created_at.tzinfo is not None` after a re-fetch; instead either (a) check the column type's `.timezone is True` flag on `Model.__table__.c["created_at"].type`, or (b) read the value back from the original in-session object before commit (with `expire_on_commit=False`). Postgres preserves tz natively, so this is a SQLite-only quirk that won't bite production but will bite test assertions.
  - `session.flush()` inside a helper triggers INSERTs immediately — for "second-insert-fails-with-IntegrityError" tests, place `session.add(...)` directly inside the `pytest.raises(IntegrityError)` block rather than calling a helper that flushes, otherwise the helper's flush raises *before* the `await session.commit()` line and pytest.raises misses it.
  - `Numeric(asdecimal=False)` is the right way to get a `Mapped[float]` for PRD's "NUMERIC" columns (temperature, top_p) without coupling the rest of the codebase to `decimal.Decimal`. Default `Numeric()` returns `Decimal`, which mypy + downstream JSON serialization both find awkward.
  - The `Uuid` type from `sqlalchemy` (capital U, 2.x) is the portable choice — stores as native `uuid` on Postgres and as CHAR(32) on SQLite, and produces a Python `uuid.UUID` on read. Pair with `default=uuid.uuid4` (the function reference, not `uuid.uuid4()` the call) so each insert generates a fresh id without sharing state across rows.
  - `expire_on_commit=False` on the session factory keeps Python-side attribute values intact after `commit()`, which is what we want for tests that inspect the just-committed object. With `expire_on_commit=True` (the default), every attribute access after commit triggers a refresh from the DB — slow, surprising, and incompatible with `await session.commit()` followed by sync attribute reads.

To test from pytest, write **sync** `def test_...` functions — calling `command.upgrade` from inside an `async def` would attempt to nest `asyncio.run` and explode. The pytest-asyncio `asyncio_mode = "auto"` setting only auto-wraps coroutine functions, so sync tests coexist fine in `tests/db/`.
  - Resolve the DB URL inside `env.py` from `os.environ["PADRINO_DB_URL"]` directly — do NOT call `get_settings()` (which is `lru_cache`d and would freeze the URL across multiple test invocations in the same process). Falling back to `Settings(_env_file=None).padrino_db_url` keeps the default in sync with the runtime settings without coupling to the disk `.env`.
  - Use `render_as_batch=True` for SQLite migrations even if the initial revision is pure `create_table` — future revisions that alter columns will need batch mode for SQLite and forgetting to flip the flag later is a footgun. Postgres ignores `render_as_batch` so this is safe to enable unconditionally per-dialect.
  - The `[tool.ruff.lint.per-file-ignores]` block in `pyproject.toml` already covers `src/padrino/db/migrations/**` with `["E402", "F401"]`, so the `models` import in `env.py` does NOT need a `# noqa: F401` comment — adding one trips `RUF100` (unused noqa).
  - Hand-writing the initial migration (rather than `alembic revision --autogenerate`) keeps it deterministic and review-friendly. For follow-up migrations, autogenerate is fine, but always inspect the diff before committing — autogenerate misses server defaults and is sometimes confused by `Uuid` cross-dialect rendering.
