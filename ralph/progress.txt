## Codebase Patterns
(Reusable learnings discovered across wave-2 iterations. Keep this section short
and general — story-specific notes live in the iteration entries below. Patterns
inherited from wave 1 are NOT duplicated here; read the archive for those.)

- Terminal-game detection (US-049): a game is "terminal" iff `Game.status == 'COMPLETED'`. The runner sets this atomically with the `GameTerminated` event row inside `GamePersistence._persist_terminated_event`. Do NOT add new `event_type='GameTerminated'` queries — filter on `Game.status` instead. The legacy event-scan pattern is gone from `padrino.api` and `padrino.gauntlets` and should not be reintroduced.
- `Game.terminal_result` is a JSON column shaped `{winner, reason, day_terminated}`. The separate `Game.terminal_reason` column was dropped — reason lives inside the JSON. Read it via `game.terminal_result["winner"]`.
- Game-status string values are UPPERCASE to match the existing Gauntlet convention: `CREATED` → `RUNNING` → `COMPLETED`. Story copy showed lowercase as an illustrative example; the codebase uses uppercase.
- Atomic terminal-write seam: `_persist_terminated_event(persistence, stored, state, ranked, day_terminated)` writes the event row, `Game.status='COMPLETED'`, `Game.terminal_result`, and (when applicable) every rating update inside ONE `session.begin()`. Tests assert rollback by monkeypatching `update_ratings_for_game` to raise — the game row and event row must NOT be persisted.
- Seat-backfill seam: `_persist_roles_assigned(persistence, stored)` writes the `RolesAssigned` event row and every `game_seats` row in ONE txn. Requires `GamePersistence.agent_builds` to cover EVERY seat in the payload; partial mappings silently skip seat insertion (the event row still goes through) so callers driving the runner without a per-seat build mapping don't crash. Demo gauntlets and full integration tests always supply the complete map.
- The runner forbidden-imports firewall test (`test_game_runner_does_not_import_forbidden_modules` in `tests/runner/test_game_runner.py`) blocks `random`, `secrets`, `datetime`, `time` imports from `src/padrino/runner/game_runner.py`. If a story needs wall-clock at the runner level, factor the helper into a sibling module (e.g. `padrino.runner.persistence` or similar) and import the helper — don't import the stdlib module from `game_runner.py`.
- Secret resolution (US-050): `padrino.llm.secrets.resolve_secret(ref)` is the single seam that turns `ModelProvider.auth_secret_ref` strings into credentials. Supported schemes are `env:VAR_NAME` and `file:/abs/path` ONLY (no `vault:`, no relative paths, no `~` expansion). Raise `SecretResolutionError` on every failure mode — never return `None` or `""`. The adapter resolves ONCE at `__init__` and stashes the value on `self._auth_secret`; never re-resolve per request. The admin POST /model-providers route also resolves at creation so a bad ref 422s immediately instead of 401-ing at game time.
- The `padrino.llm.litellm_adapter` AST firewall test (`test_litellm_adapter_does_not_import_db_or_random`) checks `mod.startswith(forbidden)` for forbidden module names including `"secrets"`. The stdlib module is `secrets`; our project module is `padrino.llm.secrets`. Because the prefix check is anchored at the start of the dotted name, `padrino.llm.secrets` does NOT match the `secrets` prefix — safe to import. Future stdlib `secrets` (token generation, etc.) imports from the adapter would still be caught.
- Recorded-cassette tests (US-051): use `vcrpy` (committed dev dep, ignored by mypy via the `vcr.*` override). The `live_llm` marker is registered in `pyproject.toml` and default-skipped by the `pytest_collection_modifyitems` hook in `tests/conftest.py`. Opt-in is either `--live-llm` (CLI flag) or `-m live_llm` (marker selector). Cassette helper: `vcr.VCR(record_mode="once" if PADRINO_RECORD_LLM=="1" else "none", match_on=("method","scheme","host","path"), filter_headers=[...], before_record_request=_scrub_request, before_record_response=_scrub_response)`. Loose `match_on` lets cassettes ignore differing request bodies across model parameters. The audit test `test_cassettes_have_no_secret_shaped_substrings` scans every committed yaml for `sk-...`, `sk-ant-...`, and `Bearer <token>` patterns and is unmarked so it runs on every default invocation.
- Probing litellm's transport: vcrpy 8.x intercepts the async httpx transport via `vcr.stubs.httpcore_stubs` (the older `httpx_stubs` module was renamed). Litellm dispatches to provider-specific URLs which can be observed by monkeypatching `httpx.AsyncClient.send`. Confirmed URLs as of the wave-2 commit: openai→`api.openai.com/v1/chat/completions`, anthropic→`api.anthropic.com/v1/messages`, cerebras→`api.cerebras.ai/v1/chat/completions`, deepinfra→`api.deepinfra.com/v1/openai/chat/completions`, ollama (with `ollama/<model>` prefix, NOT `ollama_chat/`)→`localhost:11434/api/generate`. The ollama provider's `/api/generate` returns `{"response": "<text>", ...}` (not the `/api/chat` shape with `message.content`).
- Canonical per-role prompts (US-052): `padrino.llm.prompts.load_canonical(ruleset_id, role_family)` returns a frozen `PromptTemplate` with `system_prompt` text loaded from bundled markdown under `src/padrino/llm/prompts/<ruleset_id>/<role_family>.md`. `iter_canonical_prompts()` yields all four; `canonical_prompts_by_role()` projects them into a `Mapping[Role, str]` (already family-resolved) that `LiteLlmAdapter.system_prompts_by_role` consumes — the adapter does NOT import a ruleset module. Build-pinned non-canonical prompts route through the legacy single `system_prompt` arg.
- Subpackage-vs-import namespace collision: `padrino.llm.prompts` is a package containing the `mini7_v1` ruleset directory; doing `from padrino.core.rulesets import mini7_v1` inside `prompts/__init__.py` makes the LOCAL `mini7_v1` attribute get clobbered the first time `importlib.resources.files("padrino.llm.prompts.mini7_v1")` imports the subpackage (Python rebinds the parent module's attribute). Workaround applied: alias the ruleset import (`as _mini7_ruleset`). If a future story adds another `prompts/<ruleset_id>/` directory, use the same alias pattern.
- `role_family_breakdown` shape (`Mapping[role_family_value, Mapping[str, float]]`) committed in `padrino.leaderboards.service._per_ab_role_family_breakdown`: keys are the `RoleFamily` enum values (DECEPTIVE / INVESTIGATIVE / PROTECTIVE / VANILLA_TOWN); each bucket carries `{games, wins, draws, losses, win_rate}` as floats (win_rate is `wins/games`, 0.0 when games == 0). Seats with a `role` string not parseable as `Role(...)` are skipped silently — defensive against historical rows the leaderboard route should never crash on.
- Hatch wheel packaging for non-Python data: `[tool.hatch.build.targets.wheel].artifacts = ["src/padrino/llm/prompts/**/*.md"]` is the right knob to force-include `.md` files alongside Python sources. `importlib.resources.files("padrino.llm.prompts.<ruleset>").joinpath("<file>.md").read_text()` works in src-mode immediately because the directory is on sys.path; the `artifacts` config matters when building the wheel for distribution.
- Retry / backoff seam (US-053): `padrino.llm.retry.with_retry(call, policy, *, sleeper, rng)` is the one place LLM provider calls go through exponential backoff. Both time and randomness are injected so adapter tests pin them; `sleeper` defaults to `asyncio.sleep` on the adapter and `rng` is a per-call `SeededRng(f'{phase}:{player_id}:{model}')` so replays reproduce the schedule. Exhaustion raises `RetryExhausted` carrying the last error; the adapter's `_call_model` catches it and returns `AdapterResult(status='provider_error', failure=LlmCallFailed(...))`. The adapter's `complete()` promotes the final status to `'exhausted'` (new `AdapterStatus` literal) when either the no-fallback path or the fallback path saw exhaustion. Tick.py inspects `result.status == 'exhausted'` and emits ONE `ActionTimedOut` event with `payload.reason='llm_exhausted'` plus `error_kind / error_message / attempts` (matches the new `llm_calls.error_kind / error_message` columns from migration 0006). Existing `'both_failed'` is preserved for the two-non-retryable case so legacy tests didn't have to flip.
- litellm exception classes must be imported from `litellm.exceptions` (NOT `import litellm; litellm.RateLimitError`) under mypy strict. The `[[tool.mypy.overrides]] module = ["litellm.*"]` block sets `ignore_missing_imports = true` but does NOT suppress `attr-defined` errors on the top-level `litellm` module's lazy exception re-exports. Use `from litellm.exceptions import RateLimitError, APIConnectionError, InternalServerError, BadRequestError, AuthenticationError` instead.
- `EventLog.events` returns a tuple of `StoredEvent` Pydantic models, not dicts. Access body fields via `event.body["event_type"]` (and `.body["payload"]`), not `event.get("event_type")`. The `body` is a dict, but the wrapper is frozen.
- Async scheduler seam (US-054): `padrino.runner.scheduler.run_scheduler(session_factory, *, concurrency, stop_event, adapter_factory, game_executor, semaphore, options, clock, sleeper)` is the single drain loop for `Gauntlet.status='PENDING'` rows. Status transitions: PENDING → RUNNING (claim + heartbeat) → COMPLETED (set `completed_at`, clear `heartbeat_at`). Crash recovery runs once at startup; stale-RUNNING means `heartbeat_at < now - heartbeat_interval_s * stale_factor` OR `heartbeat_at IS NULL`. Tests inject `game_executor` so they don't need to run full mini7 games. The `padrino.runner.game_runner` purity-firewall test is scoped only to `game_runner.py`; sibling impure modules in `padrino.runner/` are free to import `asyncio`, `datetime`, `time`.
- Cross-dialect datetime comparisons: SQLite drops tzinfo on read, Postgres preserves it. Use `padrino.db.repositories.gauntlets._aware(dt)` (or copy the pattern: `dt if dt.tzinfo is not None else dt.replace(tzinfo=UTC)`) before any comparison between a DB-read datetime and `datetime.now(UTC)`. The pattern will matter once US-057 lands a full Postgres test suite.
- Gauntlet status convention is uppercase `PENDING` / `RUNNING` / `COMPLETED`. `gauntlets_repo.create()` defaults to `'PENDING'`. The legacy `'QUEUED'` literal in the v1 API route was an outlier and is gone as of US-054. Any new states (e.g. `'FAILED'`) should follow the same uppercase convention.
- API pagination seam (US-055): `padrino.api.pagination.CursorPage[T]` is the standard envelope for every list endpoint — `{items, next_cursor, total_estimate}`. Cursors are opaque base64-encoded JSON: `encode_cursor(created_at, id)` for keyset pagination over time-ordered DB lists; `encode_index_cursor(n)` for in-memory ordering (the leaderboard). `paginate_keyset(session, stmt, *, created_at_col, id_col, limit, cursor)` is the canonical helper — it applies `(created_at, id) > (cursor_dt, cursor_id)` WHERE, ORDER BY both columns, and `LIMIT n+1` to detect a next page. Limit bounds are `MIN_LIMIT=1, DEFAULT_LIMIT=50, MAX_LIMIT=200`. Tampered cursors raise `InvalidCursorError`; routes call `invalid_cursor_error()` to surface a 400 `{detail: 'invalid_cursor'}`.
- Strict query params (US-055): unknown query parameters return 422 via the `Annotated[QueryModel, Query()]` FastAPI 0.115+ pattern where `QueryModel` is a `BaseModel` with `model_config = ConfigDict(extra='forbid')`. Per-route limit field uses `Field(default=DEFAULT_LIMIT, ge=MIN_LIMIT, le=MAX_LIMIT)` so out-of-range values 422 at validation, before the handler runs. New list endpoints follow this pattern uniformly.
- `games.created_at` was added in migration 0008 (US-055) and is the canonical ordering column for `GET /games`. The pre-existing `started_at` is "when the runner began ticking", which is `NULL` for unstarted games and therefore unsuitable as a sort key. Existing rows backfill `created_at` from `COALESCE(started_at, completed_at, CURRENT_TIMESTAMP)` before the NOT NULL constraint is added; rely on `Game.created_at` whenever you need an insertion-order key on games.
- Leaderboard route shape (US-055): `GET /leagues/{id}/leaderboard` keeps its existing `entries` field for back-compat and just adds a top-level `next_cursor` — it does NOT return a `CursorPage[T]` envelope, because the payload also carries `leaderboard_id / ruleset_id / prompt_version / rating_model` that don't fit `CursorPage`'s shape. Pagination is index-based over the already-sorted in-memory entries; cursor uses `encode_index_cursor`. `compute_leaderboard(...)` now accepts an optional `gauntlet_id` filter that narrows `_terminal_games_in_league` to one bracket.
- API-key auth seam (US-056): authentication is **opt-in** via `create_app(auth_required=True, admin_token=..., rate_limiter=...)`. With `auth_required=False` (the default) requests with no credentials get a *synthetic admin* `ApiKeyContext(id=None, scopes={admin}, via_admin_token=False)` — this keeps every existing route + test green without modification. `padrino.api.auth.get_auth_context(request, session)` is the single resolution point; per-route `Depends(require_admin)` / `Depends(require_read)` enforce scopes on top of it. Scope policy: `admin` scope satisfies any required set, otherwise an intersection is required. New write routes must add `dependencies=[Depends(require_admin)]`; new read routes use `Depends(require_read)`.
- Rate-limit seam (US-056): `padrino.api.auth.RateLimiter` is an in-process sliding-window counter (per-key deque of monotonic timestamps, 60-second window). `clock` is injectable for tests — substitute a `_FakeClock` callable returning a mutable float. The settings ceilings are split across three fields (`padrino_rate_limit_admin_per_minute`, `padrino_rate_limit_submitter_per_minute`, `padrino_rate_limit_spectator_per_minute`); the story's single-field shape was infeasible because the three defaults differ. Synthetic admin contexts and X-Padrino-Admin-Token shim contexts skip rate-limiting entirely — only api-key Bearer requests count against the bucket. Tests that need to exercise the limiter monkeypatch `padrino.api.auth._limit_for_scopes` to a low ceiling (e.g. 2).
- Raw-key handling convention (US-056): raw keys live ONLY in the POST /admin/keys response body (and once-only in the `--with-admin-key` bootstrap CLI later in US-058). Persistence is `sha256(raw_key)` in `api_keys.key_hash` UNIQUE + a 6-char `key_prefix` for display. The raw key format is `pk_<urlsafe-base64-32>` (≈ 46 chars); `api_keys_repo.hash_api_key(raw)` is the single place where the digest is computed. `hmac.compare_digest` is used only for the back-compat X-Padrino-Admin-Token check; sha256 lookups go through the UNIQUE index in O(log n) — the constant-time property comes from the DB layer, not Python. Any future endpoint that returns key metadata must NEVER expose `key_hash` or any `raw_key`-like value.

- **Learnings for future iterations:**
  - `monkeypatch.setattr(os.environ, "__getitem__", ...)` does NOT work — `os._Environ` doesn't accept attribute reassignment on bound methods. To count secret resolutions in a test, monkeypatch the seam itself: `monkeypatch.setattr(padrino.llm.litellm_adapter, "resolve_secret", counting_resolve)`. That's also why tests verify "single resolution" by patching the symbol the adapter imported, not by intercepting env reads.
  - FastAPI dep order: when a handler has both `body: Model` and `session: AsyncSession = Depends(get_session)`, the dep + body parse complete BEFORE handler body runs. So `test_admin_routes_rejected_when_no_session_factory` still 503s (dep fails first) even though the handler would 422 on an unresolvable secret — order of evaluation saves us. Don't rely on this for tests that need to assert the resolve_secret path; assert against bodies that pass dep+pydantic gates first.
  - File-permission policy is stricter than the story's literal text: we reject any of `g+r,g+w,o+r,o+w` (`stat.S_IRGRP | S_IWGRP | S_IROTH | S_IWOTH`), not just world-readable. This matches OpenSSH's `IdentityFile` convention. If a future story wants group-readable secret files (k8s Secret volumes mount mode 0o644), relax to `(S_IWGRP|S_IROTH|S_IWOTH)` — but keep the policy in one place inside `_resolve_file`.
  - Naming collision avoided: `padrino.llm.secrets` does NOT trip the existing `test_litellm_adapter_does_not_import_db_or_random` firewall, because that test does `mod.startswith("secrets")` and our dotted name starts with `padrino`. Stdlib `secrets` imports would still be caught. If a new module under `src/padrino/` ever happens to start with `secrets.` (e.g. moved to top-level), revisit the firewall.

## 2026-05-15T00:30 - US-053
- New module `padrino.llm.retry` (`RetryPolicy`, `RetryExhausted`, `RetryAttempt`, `LlmCallFailed`, `default_retry_policy`, `with_retry`, `DEFAULT_RETRY_ON`). Bounded exponential backoff: `delay = min(base * 2**idx, max_delay) * jitter` where `jitter` is a `SeededRng`-driven uniform in `[0.5, 1.0]`. Default policy retries on `litellm.exceptions.{RateLimitError, APIConnectionError, InternalServerError}` and `TimeoutError`; never on `BadRequestError` / `AuthenticationError`. On exhaustion `with_retry` raises `RetryExhausted(attempts, last_error)`; non-retryable errors short-circuit by re-raising the original exception.
- Wired into `LiteLlmAdapter`: each `_call_model` wraps `litellm.acompletion(**kwargs)` in `with_retry(...)` with a per-call `SeededRng(f'{obs.phase}:{obs.you.player_id}:{model_id}')` and an injectable `sleeper` (defaults to `asyncio.sleep`). The adapter's two new init kwargs are `retry_policy: RetryPolicy | None = None` and `sleeper: Callable | None = None`. On exhaustion the adapter returns `AdapterResult(status='provider_error', failure=LlmCallFailed(error_kind, error_message, attempts))` for the per-model path; `complete()` promotes the final status to the new `'exhausted'` literal when EITHER the no-fallback path or the fallback path saw exhaustion (`failure is not None`).
