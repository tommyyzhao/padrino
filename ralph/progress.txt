## Codebase Patterns
(Reusable learnings discovered across wave-3 iterations. Keep this section short
and general — story-specific notes live in the iteration entries below. Patterns
inherited from waves 1-2 are NOT duplicated here; read the archive for those.)

- Markdown-fence quirk on Cerebras GLM-4.7: chat-tuned models routinely wrap their JSON output in ```json ... ``` fences. `padrino.llm.litellm_adapter._strip_code_fence` normalizes this in the impure adapter layer before `parse_agent_response` runs. If a new provider exhibits a different wrapper (e.g. `<json>...</json>` tags, leading "Here is my response:" preamble), extend `_strip_code_fence` AND add a unit test in `tests/llm/test_litellm_adapter.py::test_markdown_code_fence_is_stripped_before_parse`. Do NOT loosen `parse_agent_response` in pure-core — its contract is "give me clean JSON or be rejected".
- Cassette re-recording (`PADRINO_RECORD_LLM=1`): vcrpy's `record_mode="once"` only writes a cassette if the file is MISSING — `rm` the existing yaml first or the run silently no-ops. Recording one provider's `canonical_response.yaml` will typically end up with 3 interactions per cassette because litellm's `_extract_text` raises on gzipped-bytes during the live run (the body decodes fine at REPLAY time), triggering 2 retries. Replay only consumes the first matching interaction, so the extras are dead weight but harmless. Adapter timeout defaults to 5 s; the contract-test fixture extends it to 60 s while `PADRINO_RECORD_LLM=1` so DeepInfra's cold start (15+ s on DeepSeek-V4-Flash) can land. Synthetic-canonical providers (no API key in `.env`) get `ProviderCase(synthetic_canonical=True)` plus a `TODO(US-072)` comment; the parametrize entry stays in the live_llm marker set but skips at runtime.
- The `>=70%` parse-rate quality gate in `tests/integration/test_real_providers.py` exists because the original test passed despite 100% of responses failing to parse (the runner silently coerced them to NOOP / ABSTAIN). Any new real-LLM integration test MUST carry forward this gate or an equivalent — `raw_response is non-empty` is NOT a sufficient assertion.
- `padrino.core.observation_privacy.assert_ranked_observation_safe` enforces the privacy firewall at runtime per game. Wave 3 adds `audit_observation_log_for_seat` (US-078) as the offline auditor that returns findings instead of raising — runtime check and audit check share their forbidden-field list, do NOT drift the two lists out of sync.
- Migration sequencing tests live in TWO files: `tests/db/test_migrations.py` (SQLite-only v1 subset) and `tests/db/test_migrations_all_dialects.py` (SQLite + Postgres via testcontainers). The all-dialects file's `EXPECTED_TABLES` set drives the Postgres fixture's between-test DROP-cascade cleanup; whenever a new table lands in a migration, add it to `EXPECTED_TABLES` in `test_migrations_all_dialects.py` or the full-cycle Postgres test will fail "relation already exists" on the second upgrade head. The SQLite-only file's `EXPECTED_TABLES` is checked as a SUBSET only, so adding tables there is optional.
- Streaming size caps in FastAPI must be enforced at the stream layer, not only after `await request.body()` completes. `Request.stream()` yields chunks as they arrive; accumulating into a list and bailing once the total exceeds the cap is the defensive read path. Honor `Content-Length` up front when present (short-circuit before reading any bytes) AND keep the streamed bound for the chunked-transfer / lying-Content-Length case. Without both, a malicious submitter can force the server to buffer the entire payload before the post-buffer length check fires.
- httpx + ASGITransport supports two body shapes for testing oversized payloads: a `bytes` body with an explicit oversized `Content-Length` header tests the up-front short-circuit, while an async iterator (`AsyncIterable[bytes]`) tests the streamed bound because httpx switches to chunked transfer (no Content-Length) when the body is iterable. Both should converge on the same 413 response.

## 2026-05-20 — Wave 3 kickoff (post-Wave-2 audit)

Wave 2 self-reported 22/22 stories passing, but a post-completion audit found:
1. The `test_real_providers_one_game_day` integration test had never been run.
2. When run, 100% of Cerebras `zai-glm-4.7` responses returned `status=invalid_json` because the adapter did not strip the markdown ```json ... ``` fence the model consistently emits.
3. The runner silently coerced every failing response to a safe-fallback (NOOP / ABSTAIN), so the game appeared to run end-to-end while no real model behavior was being measured.
4. The integration test's assertion (`raw_response or status in _FAILURE_STATUSES`) did not catch this because `invalid_json` was not in the failure statuses set and `raw_response` was always non-empty.

Fix landed pre-Wave-3 on `ralph/padrino-v2` as commit `4e4ed22` and merged forward to `main` and `ralph/padrino-v3`:
- `padrino.llm.litellm_adapter._strip_code_fence` normalizes ```...``` wrappers before parsing.
- 5 new unit tests in `tests/llm/test_litellm_adapter.py` (4 fence variants + unfenced-passthrough).
- A new `>=70%` parse-rate quality gate in `tests/integration/test_real_providers.py` so this class of silent-coercion regression cannot recur.
- Cost cap bumped $0.10 -> $0.20 because well-formed responses are longer than truncated failures.

After the fix: 25/25 calls return `status=ok` against the real Cerebras + DeepInfra providers. All four backend gates remain green; coverage on core/ is 99%.

Wave 3 scope (8 stories, US-071..US-078):

- **Learnings for future iterations:**
  - The bundle's hash chain excludes `event_hash`, `prev_event_hash`, and `created_at` from the hashed body — but `terminal_result` lives at the TOP level of the bundle, NOT inside any event, so it is NOT covered by the hash chain. An attacker can edit `bundle.terminal_result` independently of the event log without invalidating any event hash. The new `inconsistent_terminal` check exists exactly because the hash chain alone does not protect bundle-level fields; any future bundle-level claim (e.g. `bundle.signer_label`) needs its own consistency check or it must be moved into a hashed event.
  - `EXPORT_FORBIDDEN_PAYLOAD_KEYS` and `PUBLIC_TRANSCRIPT_FORBIDDEN_KEYS` are intentionally DIFFERENT sets. Export drops model/provider/rating provenance because the runner should never emit those into the event log. Public transcript additionally drops `role` and `faction` because those legitimately appear in `RolesAssigned.payload.assignments` but must not surface publicly for a non-terminal game. Do not collapse the two sets — the export guard would then reject every legitimate bundle, and the public guard would be missing the provenance keys.
  - The auth pipeline runs rate-limiter BEFORE the route body executes, so even a 413 (raised inside the route) still consumes one rate-limit token. This is what makes the sybil-burst test cheap: posting empty bodies with an oversized Content-Length header is enough to drive the rate limiter to exhaustion without going through the full bundle parse path. If a future story moves the size check upstream of auth (e.g. into a middleware), the sybil test would need to switch to a different request that actually authenticates.
  - To bound a streamed request body in Starlette/FastAPI: iterate `request.stream()` and break (raise HTTPException) once the running total exceeds the cap. Do NOT rely on Starlette's built-in body size limit (configurable on Uvicorn, but not on TestClient/ASGITransport — and the default is generous). The route-level bound is the only portable defense.
