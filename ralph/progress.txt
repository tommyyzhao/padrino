## 2026-05-22 — Wave 4 kickoff (post-Wave-3 verification)

Wave 3 finished with all 8 stories self-reporting `passes: true`, but per the
established discipline this got an auditor's run before being declared truly
done. The audit found:

1. `test_real_providers_full_game.py` (US-071) had never actually been run.
   First execution exposed a counting bug in the parse-rate quality gate:
   `parse_rate = ok_count / total` excluded `fallback_ok`, treating a healthy
   primary→fallback path (Cerebras failed, DeepInfra fallback returned valid
   JSON) as a parse failure. Real run: 40 ok + 31 fallback_ok = 71/71
   parseable, gate reported 56% and failed.
2. The same test's "every seat must produce a non-NOOP / non-ABSTAIN action"
   assertion was unachievable in legitimate games: a villager who ABSTAINs
   on Day 1 and dies Night 1 has no further chances (Day Discussion forces
   NOOP, villagers cannot inspect/protect/kill).

Fixed pre-Wave-4 as commit `884076c`:
- New `_PARSED_OK_STATUSES = {"ok", "fallback_ok"}` constant used across
  all three real-provider integration tests (day-1 sanity, full game,
  mini-gauntlet).
- Action-coverage assertion loosened from "every seat" to "at least
  len(seats) - 2 seats" (5 of 7), preserving the silent-coercion detection
  intent while tolerating early-game eliminations.

Real-run evidence post-fix:
- US-071: winner=MAFIA reason=PARITY_REACHED calls=70 parse=100% cost=$0.66
- US-073: games=3 winners=[MAFIA,MAFIA,MAFIA] calls=210 parse=100% cost=$3.08

Total real-money cost across the verification: $3.74.

The three-MAFIA-wins sample in US-073 is too small to draw conclusions
(Wilson CI on 0/3 TOWN wins is (0.0, 0.56)), but it does suggest the
current single-model lineup (all 7 seats on Cerebras GLM-4.7) may have a
prompt-driven faction imbalance. Wave 4 addresses this by adding model
diversity: Z.AI GLM-5.1, DeepInfra Gemma-4-26B-A4B-it, Xiaomi Mimo-v2.5
+ Mimo-v2.5-pro, plus Z.AI GLM-4.7 as a same-model multi-host fallback
for Cerebras (operational reliability) — and US-083 runs a heterogeneous
gauntlet with one distinct model per seat so the leaderboard reflects
head-to-head competition, not self-play.

Wave 4 scope (5 stories, US-079..US-083):
- US-079: Same-model multi-host fallback in RoutingPolicy + Z.AI GLM-4.7
  as Cerebras's same-model fallback (no leaderboard bifurcation)

Dependencies: US-080 → US-079 (Z.AI must be registerable before it can be

## 2026-05-22 — US-079 (same-model multi-host fallback)

Landed by maintainer (not Ralph). The architectural slice that makes the
rest of wave-4 possible.

What shipped:
- `RoutingPolicy.same_model_hosts: tuple[SameModelHost, ...]` — alternate
  provider endpoints serving the SAME upstream weights. Each carries
  `provider`, `litellm_model_id`, `auth_secret_ref`, and optional
  `api_base`.
- `LiteLlmAdapter` now iterates primary → same_model_hosts → fallback.
  Per-host retry budgets are isolated (a primary exhausting its 3
  attempts does NOT consume any same-model host's budget). Per-host
  `api_base` and `api_key` are forwarded to `litellm.acompletion` only
  when set, so the primary call keeps its pre-US-079 kwargs bit-for-bit.
- New `AdapterStatus` literal `same_model_fallback_ok`. Added to
  `_PARSED_OK_STATUSES` in all three real-provider integration tests.
- `Settings.build_routing_policy()` injects the canonical Z.AI GLM-4.7
  same-model host whenever (`padrino_cerebras_zai_glm47_zai_fallback`
  AND `zai_api_key` AND `primary_model == "cerebras/zai-glm-4.7"`).
- `padrino_zai_api_base` defaults to the Z.AI **Coding Plan** endpoint
  `https://api.z.ai/api/coding/paas/v4` (NOT the General API
  `https://api.z.ai/api/paas/v4`). The Coding Plan is what our paid
  subscription serves; using the General API while on the Coding Plan
  returns a 429 dressed up as "Insufficient balance or no resource
  package".
- Leaderboard non-bifurcation invariant asserted at
  `tests/api/test_public_models.py::test_same_model_fallback_does_not_bifurcate_leaderboard_row`:
  even when llm_calls carry the new status, `rollup_by_model` returns
  exactly one row per `(provider, model_name, model_version)` of the
  AgentBuild.

Real-provider verification:
- `tests/integration/test_same_model_fallback.py` — BOTH tests passed:
  * Primary healthy: Cerebras GLM-4.7 served cleanly; Z.AI never touched.
  * Forced fallback: monkeypatched primary to raise RateLimitError; Z.AI
    GLM-4.7 (Coding Plan endpoint) served the same-model attempt with
    valid JSON. Final status: `same_model_fallback_ok`.

Gates: ruff clean, ruff format clean, mypy strict clean, 1212 unit
tests pass, core coverage 99%.

Carry-forward discipline for the rest of wave-4:
- US-080's GLM-5.1 cassette recording will hit the **Coding Plan**
  endpoint; ensure the cassette records `https://api.z.ai/api/coding/paas/v4`
  in the request URL. The secret scrub should still catch the ZAI key
  prefix per US-080 AC.
- The "insufficient balance" -> RateLimitError shape from Z.AI is
  surfacing as a real signal — when iterating, this likely indicates
  the wrong base_url, not the wrong key.

### Learnings for future iterations

- **`ModelProvider.base_url` is the api_base column.** Wave-4 stories
  US-079 / US-080's AC text says "api_base=..." in places — that means
  set `ModelProvider.base_url`. Do NOT introduce a parallel `api_base`
  column. Per-model dispatch identifier overrides go on
  `ModelConfig.litellm_model_id` (US-082, migration 0013).
- **`<promise>COMPLETE</promise>` is a hair-trigger.** Codex's first
  Ralph-Codex iteration emitted COMPLETE after US-082 while three
  stories (US-080, US-081, US-083) were still `passes: false`. The
  prompt was unambiguous about the stop condition but Codex skipped
  the `passes:false` count. Prompt patched: the iteration MUST now
  run `jq '[.userStories[] | select(.passes==false)] | length' \
  ralph/prd.json`, quote the output in its reply, and only emit
  COMPLETE when that output is `0`.
- **`./ralph/ralph-codex.sh` archive logic is destructive when
  `.last-branch` lags behind a manual `branchName` edit.** Today's
  symptom: the script copied the current wave-4 `prd.json` /
  `progress.txt` ON TOP OF the already-populated
  `ralph/archive/2026-05-22-padrino-v3/` folder (overwriting the
  archived wave-3 content) and reset `ralph/progress.txt` to an
  empty stub. Both `ralph.sh` and `ralph-codex.sh` patched: the
  archive copy + reset is now skipped when the destination folder
  already has `prd.json` or `progress.txt` (manual archive wins).
- **Bootstrap YAML now supports inline model_configs.** The
  `ProviderSpec.models: list[ProviderModelSpec]` field lets a single
  bootstrap step seed both `model_providers` and `model_configs`
  rows. US-080 (Z.AI GLM-5.1) should use this shape rather than
  bootstrap-then-manual-INSERT for model_configs.
